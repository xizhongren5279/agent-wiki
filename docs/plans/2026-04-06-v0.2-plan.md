## 一、多格式摄入

### 改造 wiki_ingest

当前：只接受 `.md` 文件，其他格式跳过。

改后：

| 格式 | 插件处理方式 | Agent 做什么 |
|------|------------|------------|
| `.md` | 直接复制到 raw/ | — |
| `.txt` | 改后缀为 .md，复制到 raw/ | — |
| `.html` | 正则去标签提取正文，转 .md 存到 raw/ | — |
| `.csv` | 读取内容转 markdown 表格，存为 .md 到 raw/ | — |
| `.xlsx` / `.xls` | 原样存到 raw/，返回文件路径 | Agent 多模态读取 |
| `.pdf` | 原样存到 raw/，返回文件路径 | Agent 多模态读取 |
| `.docx` | 原样存到 raw/，返回文件路径 | Agent 自己处理 |
| 图片（.png/.jpg/.jpeg/.gif/.webp） | 存到 raw/assets/，返回路径 | Agent 直接看 |

### 返回格式变更

```json
{
  "status": "ok",
  "ingested": 3,
  "total": 5,
  "files": [
    {
      "name": "article.md",
      "type": "markdown",
      "content_preview": "前 500 字..."
    },
    {
      "name": "data.csv",
      "type": "converted",
      "converted_from": "csv",
      "content_preview": "| col1 | col2 | ..."
    },
    {
      "name": "report.pdf",
      "type": "binary",
      "note": "原文件已存储，Agent 需直接读取 raw/report.pdf"
    }
  ]
}
```

### CSV → Markdown 表格转换逻辑

```
1. 读取 CSV 文件
2. 第一行作为表头
3. 转为 markdown 表格格式：| col1 | col2 |
4. 添加分隔行：| --- | --- |
5. 逐行转为表格行
6. 写为 .md 文件
```

### HTML → Markdown 转换逻辑

```
1. 读取 HTML 文件
2. 去除 <script>、<style> 标签及内容
3. 去除所有 HTML 标签，保留文本
4. 清理多余空行
5. 写为 .md 文件
```

## 二、Query 反馈（用户确认版）

### 新增工具：wiki_feedback

**核心理念**：用户问问题 → Agent 回答 → 有价值的分析不应该蒸发在聊天记录里 → 用户确认后存入 Wiki。

**与 Karpathy 原文的对齐**：
> "good answers can be filed back into the wiki as new pages"

**风险控制**：
- ❌ 不自动存：防止幻觉复利（错误答案写入 Wiki → 下次搜到当作事实 → 错越滚越大）
- ✅ 用户说"存"才存
- ✅ 标记来源为 `type: query-derived`，区分原始材料编译 vs 问答生成

### 工具定义

```
wiki_feedback
  入参：
    - question: string   — 用户原始问题
    - answer: string      — Agent 的回答内容
    - sources: string[]   — 参考了哪些 Wiki 页面
    - tags: string[]      — 标签
    - category: string    — 分类
  返回：
    - status: "ok"
    - file: "wiki/反馈-标题.md"
    - type: "query-derived"
```

### 生成的 Wiki 页面格式

```markdown
---
title: "Q&A: 用户的原始问题"
date: 2026-04-06
tags: [标签1, 标签2]
category: 分类名
type: query-derived
sources:
  - "[[wiki/相关页面1.md]]"
  - "[[wiki/相关页面2.md]]"
---

# Q&A: 用户的原始问题

## 问题
用户的原始问题文本

## 回答
Agent 的回答内容

## 参考来源
- [[相关页面1]]
- [[相关页面2]]

---
*来源: 用户问答 · 生成时间: 2026-04-06 10:35*
```

### SCHEMA.md 新增操作指南

```markdown
## Agent 操作指南（追加）

- 用户问完问题后，如果回答中有综合分析、对比、洞察等有价值内容，
  主动问用户："这段分析不错，要存到知识库吗？"
- 用户确认后，调用 wiki_feedback 保存
- 保存时必须标记 type: query-derived，让用户区分哪些是原始材料、哪些是问答生成
```

### 防止循环引用

Wiki 页面的 `type` 字段有两个值：
- `compiled`：来自原始材料的编译（可信度高）
- `query-derived`：来自问答反馈（需要人工确认）

Agent 搜索时，`compiled` 类型的结果排在 `query-derived` 前面。

## 三、交叉关联标记（只标记不动手）

### 改造 wiki_compile

当前：编译完就结束。

改后：编译完额外搜索已有 Wiki 页面，找到关联的页面，在 Wiki 页面底部添加"相关页面"区块。

**重要约束**：
- ❌ 不自动修改已有页面的正文内容（防止错误传播）
- ❌ 不自动覆盖用户手动编辑的内容
- ✅ 只在当前编译的页面底部添加 `## 相关页面` 链接列表

### Wiki 页面新增区块

```markdown
## 相关页面
- [[RAG系统对比]] (标签重叠: RAG, 向量检索)
- [[知识管理方案]] (同分类: AI工程化)
```

### 关联逻辑（纯标签+分类匹配）

```
1. 获取当前页面的 tags 和 category
2. 遍历所有已有 Wiki 页面
3. 标签重叠 ≥ 2 个 → 标记为关联
4. 同 category → 标记为关联
5. 按重叠度排序，取 top 5
```

## 四、lint 溯源检测

### state.json 增加哈希字段

```json
{
  "ingested": ["article.md"],
  "tagged": ["article.md"],
  "compiled": [
    {
      "source": "article.md",
      "hash": "sha256:abc123...",
      "compiled_at": "2026-04-06T10:35:00"
    }
  ]
}
```

### wiki_compile 改动

编译时计算源文件的 SHA-256 哈希，存入 state.json。

### wiki_lint 新增检测项

```
1. 读取 state.json 中的 compiled 列表
2. 对每个已编译文件，计算当前哈希
3. 与编译时哈希比对
4. 不一致 → 标记为 warning："源文件已变更，Wiki 页面可能过期"
5. 不自动重新编译（等用户决定）
```

### 新增 issue 类型

```json
{
  "type": "source_changed",
  "severity": "warning",
  "source": "RAG系统工程化实践.md",
  "detail": "源文件 raw/article.md 已变更（编译时哈希: abc123, 当前哈希: def456），Wiki 页面可能过期"
}
```

## 五、SCHEMA.md 可定制化

### wiki_init 增加参数

```typescript
wiki_init({
  dir: string,
  topics: string[],
  interests: string[],
  // 新增参数
  writing_style?: "微信公众号" | "知乎专栏" | "博客" | "学术论文",
  tag_language?: "中文" | "英文" | "混合",
  auto_compile?: boolean  // 是否摄入后自动提示编译
})
```

### SCHEMA.md 的写作规则部分根据 writing_style 动态生成

每种写作风格对应不同的规则模板：

| 风格 | 规则差异 |
|------|---------|
| 微信公众号 | 短句、加粗关键句、开头抓人、不超 3000 字 |
| 知乎专栏 | 更长的分析、允许专业术语、可以有代码块 |
| 博客 | 自由风格、可以技术向 |
| 学术论文 | 严谨引用、有摘要和参考文献 |

## 实施优先级

```
Phase 1（P0）：多格式摄入 + Query 反馈
Phase 2（P1）：交叉关联标记 + lint 溯源 + SCHEMA.md 可定制化
```

每个 Phase 预计一个会话完成。
